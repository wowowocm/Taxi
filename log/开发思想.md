## 一、项目开发思想

### 1.1 核心理念：从数据到洞察的全链路

本项目的开发遵循 **"原始数据 → 清洗加工 → 分析挖掘 → 可视化呈现 → 实时监控"** 的五层数据价值递进模型：

```
┌──────────────────────────────────────────────────────────────┐
│                    数据价值递进金字塔                         │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│   L5  实时监控    🖥 实时运行态势大屏 (WebSocket/轮询)        │
│                   决策者实时掌握城市脉动                       │
│                                                              │
│   L4  可视化      📊 Web交互看板 + 静态图表 + 热力地图        │
│                   让数据变得可见、可交互、可理解               │
│                                                              │
│   L3  分析挖掘    🔬 时序分析 + 空间分析 + DBSCAN聚类         │
│                   从清洗后的数据中提取出行规律                 │
│                                                              │
│   L2  数据清洗    🧹 5条规则流水线 (坐标→速度→切片→OD→过滤)  │
│                   将原始GPS信号转化为结构化OD数据              │
│                                                              │
│   L1  原始数据    📥 160万条GPS轨迹 (500辆出租车 × 全天候)    │
│                   深圳市真实出租车运营数据                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 架构设计原则

| 原则 | 说明 | 体现在项目中的具体做法 |
|------|------|----------------------|
| **模块化分离** | 各功能独立为模块，单一职责 | `src/` 下 7 个独立 .py 模块，每个模块职责明确 |
| **配置驱动** | 所有阈值、路径、参数集中在配置文件中 | `src/config.py` 统一管理 25+ 配置项 |
| **渐进式构建** | 从核心能力出发，逐步扩展 | 先有 Python 分析脚本 → Matplotlib 静态图表 → Flask Web → MySQL + 实时大屏 |
| **降级容错** | 系统任一组件的失败不波及其他 | 前端每个 API 独立 fetch + `catch → null`，图表独立渲染 |
| **可迁移部署** | 宿主机开发 → 虚拟机部署 | `setup_win.bat` / `setup_vm.sh` 双平台一键安装 |
| **日志可追溯** | 每次修改、每次Bug修复都有记录 | `/log` 文件夹按日期+主题命名，README 汇总索引 |

### 1.3 为什么选择这个技术栈

| 决策 | 选择 | 理由 |
|------|------|------|
| **后端框架** | Flask | 轻量、灵活，适合数据分析场景；Python 原生生态兼容性好 |
| **数据库** | MySQL | 项目要求部署在 CentOS VM，MySQL 8.0 已预装 |
| **缓存表设计** | realtime_stats (MySQL 物化视图) | 避免每次 API 请求都执行聚合查询，KPI/热力图/排行预计算 |
| **前端地图** | Leaflet (实时大屏) + ECharts geo 系列 (分析看板) | Leaflet 免费、轻量、热力图插件成熟；ECharts geo 与项目统一主题 |
| **图表库** | ECharts 5.5 | 功能全面、中文友好、dark/shine 双主题内建、支持 map/graph 地理系列 |
| **Python 版本** | 3.8.10 (主) + 3.12.4 (备) | 3.8 兼容 CentOS 7 依赖；3.12 最新特性备用 |
| **数据处理** | Pandas + NumPy | 业界标准，CSV 读写、分组聚合、统计计算一站式 |
| **聚类算法** | DBSCAN (sklearn) | 无需预设聚类数，天然处理噪声点，适合 GPS 密度分布不均匀的场景 |

### 1.4 数据驱动开发 (DDD) 流程

每次功能迭代都遵循以下流程：

```
1. 理解数据 → 2. 编写分析模块 → 3. 编写可视化 → 4. 暴露 API → 5. 前端集成 → 6. 记录日志
```

这与传统的前后端分离开发不同——**数据是项目的第一公民**。每个 API 端点都由 `web_app.py` 中的私有函数（如 `_get_kpis()`）封装，确保业务逻辑与路由层解耦。

## 四、关键技术实现细节

### 4.1 数据清洗：5条规则流水线

```python
# 洗清效率: 1,601,307 → 8,517 (压缩比 188:1)

规则1: 坐标范围过滤
  if (lng < 113.5 or lng > 114.5 or lat < 22.4 or lat > 22.9): 丢弃
  效果: 1,978 条异常 → 1,599,329

规则2: 速度过滤
  if (speed < 0 or speed > 150): 标记异常

规则3: 载客行程切片
  按 OpenStatus 状态变化 (1→0→1) 识别每次载客行程的起止点
  效果: 1,599,329 个GPS点 → 16,783 段行程

规则4: 行程OD提取
  每段行程提取: 起点(SLng,SLat) + 终点(ELng,ELat) + 时长 + 距离 + 均速

规则5: 异常行程过滤
  if (duration < 1min or duration > 240min): 丢弃
  if (distance < 0.2km): 丢弃
  if (avg_speed < 5 or avg_speed > 100): 丢弃
  效果: 16,783 → 8,517
```

### 4.2 核心算法：DBSCAN 空间聚类

| 参数 | 值 | 含义 |
|------|-----|------|
| `eps` | 0.005° | 聚类邻域半径 (约500米) |
| `min_samples` | 20 | 核心点最小邻居数 |
| 输入 | 8,517 个起点坐标 | — |
| 输出 | 28 个聚类 + 6,583 个噪声点 (77.3%) | 噪声点离散分布，符合出租车随机出行特性 |
| 最大聚类 | 4,092 点 | 位于 (114.083, 22.552) — 深圳福田/罗湖市中心 |

### 4.3 空间网格OD矩阵

```
深圳地理范围: 113.5°~114.5° × 22.4°~22.9°
网格划分: 101列 × 56行 = 5,656 个网格单元
网格大小: ~1km × 1km (0.01° × 0.009°)

OD矩阵: 5,656 × 5,656 稀疏矩阵
  · 总出行量: 8,517
  · 网格内部出行 (对角线): 1,530 (18.0%) — 同网格内起止
  · 跨网格出行: 6,987 (82.0%)    — 跨区域出行
```

### 4.6 Flask 服务的数据预加载策略

```
启动流程 (main.py --web):

  2. init_data() — 一次性预加载:
     ├─ 读取 CSV (8,517 条 → pandas DataFrame)
     ├─ 初始化 TemporalAnalyzer (计算24h分布、高峰、效率)
     └─ 初始化 SpatialAnalyzer (构建网格、DBSCAN、OD矩阵、净流量)
  3. app.run() 启动 HTTP 服务
```
