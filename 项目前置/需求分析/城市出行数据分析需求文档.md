## 1. 项目概述

### 1.1 项目背景

随着智慧城市建设的深入推进，城市交通管理面临日益复杂的挑战。出租车作为城市公共交通的重要组成部分，其运营数据蕴含丰富的城市出行规律信息。本项目基于深圳市500辆出租车全天候GPS轨迹数据，通过数据清洗、特征工程与可视化分析，深入挖掘城市居民出行模式，为交通管理部门提供数据驱动的决策支持。

### 1.2 项目目标

| 目标层次 | 具体目标 | 衡量标准 |
|:---|:---|:---|
| **核心目标** | 构建出租车出行行为分析体系 | 完成全部分析模块开发 |
| **数据目标** | 将原始GPS点数据转化为结构化出行记录 | 数据清洗通过率 ≥ 95% |
| **分析目标** | 揭示城市出行时空规律与热点区域 | 产出 ≥ 5 个维度的分析结论 |
| **应用目标** | 为交通调度与城市规划提供决策依据 | 形成可落地的优化建议 |

### 1.3 项目范围

- **研究区域**：广东省深圳市（经度 113.5°E–114.5°E，纬度 22.4°N–22.9°N）
- **数据规模**：500 辆出租车 × 全天候 GPS 轨迹 ≈ 160 万条原始记录
- **分析周期**：单日（00:00:00 – 23:59:59）
- **分析对象**：出租车载客出行行为（不含空驶巡游）

### 1.4 术语定义

| 术语 | 英文 | 定义 |
|:---|:---|:---|
| GPS轨迹点 | GPS Trace Point | 单个时间戳下的车辆位置、速度、载客状态快照 |
| 出行记录 | Trip Record | 一次完整的载客行程，包含起点、终点、起止时间 |
| 载客状态 | Occupancy Status | OpenStatus=0 表示载客，OpenStatus=1 表示空车 |
| 时空热力 | Spatiotemporal Heatmap | 基于地理坐标和时间维度的密度分布可视化 |
| OD矩阵 | Origin-Destination Matrix | 区域间出行起止点的流量矩阵 |

## 3. 数据清洗需求

### 3.1 清洗流程总览

```
原始GPS数据                         清洗后出行数据
    │                                    ▲
    ▼                                    │
┌─────────┐   ┌──────────┐   ┌───────────────┐   ┌──────────┐
│ 异常值  │──▶│ 坐标范围 │──▶│ 载客状态切片  │──▶│ OD提取  │
│ 检测    │   │ 过滤     │   │ (连续Open=0)  │   │ 与合并  │
└─────────┘   └──────────┘   └───────────────┘   └──────────┘
                                                    │
                                              ┌──────────┐
                                              │ 异常行程 │
                                              │ 过滤     │
                                              └──────────┘
```

### 3.2 详细清洗规则

#### 规则1：异常坐标过滤

| 项目 | 内容 |
|:---|:---|
| **规则描述** | 过滤超出深圳市地理范围的GPS坐标 |
| **过滤条件** | `Lng < 113.5 OR Lng > 114.5 OR Lat < 22.4 OR Lat > 22.9` |
| **处理方式** | 直接剔除 |
| **预计影响** | ≈ 0.1% 记录被剔除（约 1,978 条） |

#### 规则2：异常速度处理

| 项目 | 内容 |
|:---|:---|
| **规则描述** | 识别并处理不合理速度值（如 >150 km/h 的城市道路速度） |
| **过滤条件** | `Speed < 0 OR Speed > 150` |
| **处理方式** | 标记为异常，以相邻有效值线性插值填充或剔除 |
| **预计影响** | 少量极端值 |

#### 规则3：载客行程切片

| 项目 | 内容 |
|:---|:---|
| **规则描述** | 按车辆分组，根据 OpenStatus 字段识别连续载客段落 |
| **核心逻辑** | `OpenStatus` 从 1→0 切换时标记行程开始，从 0→1 切换时标记行程结束 |
| **特殊处理** | 数据首尾的未闭合行程（无起始或无结束标记）予以丢弃 |

#### 规则4：行程OD提取

| 项目 | 内容 |
|:---|:---|
| **规则描述** | 提取每个行程片段的起止时间、起止坐标 |
| **输出字段** | VehicleNum, Stime, SLng, SLat, ELng, ELat, Etime |

#### 规则5：异常行程过滤

| 规则 | 阈值 | 理由 |
|:---|:---|:---|
| 最小行程时长 | ≥ 1 分钟 | 过滤GPS误触发产生的"伪行程" |
| 最大行程时长 | ≤ 4 小时 | 排除设备故障导致的异常长记录 |
| 最小行程距离 | ≥ 200 米 | 排除原地摆动产生的虚假OD |
| 平均速度合理性 | 5–100 km/h | 排除GPS漂移或设备异常 |

### 3.3 清洗质量验收标准

| 验收项 | 标准 | 验证方法 |
|:---|:---|:---|
| 坐标有效率 | ≥ 99.5% | 统计深圳区域内坐标占比 |
| 行程提取完整率 | ≥ 95% | 对比原始载客段落与提取行程数 |
| 异常行程率 | ≤ 3% | 统计被规则5过滤的行程比例 |
| OD分布合理性 | 符合城市路网 | 将OD点叠加地图进行可视化校验 |

### 4.1 功能模块总览

```
┌─────────────────────────────────────────────────────────────────┐
│                   城市出行数据分析系统                            │
├─────────────┬─────────────┬──────────────┬──────────────────────┤

├─────────────┼─────────────┼──────────────┼──────────────────────┤
```

### 4.2 模块一：数据预处理

#### FR-1.1 数据导入

- **需求描述**：支持导入CSV格式的原始GPS轨迹数据，自动识别字段类型和编码
- **输入**：`乘车出行数据源.csv` 格式的原始数据文件
- **输出**：标准化的内存数据结构（DataFrame）
- **异常处理**：文件格式校验失败时给出明确错误提示

#### FR-1.2 数据质量报告

- **需求描述**：自动生成数据质量评估报告，包含以下内容：
  - 总记录数、车辆数、时间跨度
  - 各字段缺失值统计
  - 异常坐标分布与占比
  - 速度分布直方图
  - 载客/空车状态比例
- **输出格式**：控制台打印 + Markdown文件

#### FR-1.3 自动清洗流水线

- **需求描述**：按照第3章定义的清洗规则，构建自动化数据清洗流水线
- **处理步骤**：异常值检测 → 坐标过滤 → 行程切片 → OD提取 → 异常行程过滤
- **进度反馈**：每步处理完成后输出处理记录数和耗时

#### FR-1.4 清洗结果导出

- **需求描述**：将清洗后的出行OD数据导出为CSV文件
- **输出格式**：`taxi_sz.csv`（VehicleNum, Stime, SLng, SLat, ELng, ELat, Etime）
- **编码**：UTF-8 with BOM（兼容Excel中文显示）

### 4.3 模块二：时空分析

#### FR-2.1 出行热力分布

- **需求描述**：基于时间维度，生成不同时段的出行起止点热力图
- **时段划分**：
  - 早高峰：07:00–09:00
  - 日间平峰：09:00–17:00
  - 晚高峰：17:00–19:00
  - 夜间：19:00–07:00（次日）
- **可视化形式**：基于地图底图的热力图层，颜色深度表示打车需求量

#### FR-2.2 区域OD矩阵

- **需求描述**：将深圳市划分为网格区域（建议 1km×1km），构建区域间出行OD矩阵
- **输出**：N×N 矩阵，M[i][j] 表示从区域 i 到区域 j 的出行量
- **用途**：识别城市主要通勤走廊和区域联系强度

#### FR-2.3 出行高峰识别

- **需求描述**：以小时为单位统计出行量时间分布，自动识别早晚高峰时段
- **统计维度**：
  - 小时级出行量变化曲线
  - 载客率时间变化（载客车辆数/总活跃车辆数）
  - 平均行程时长随时间变化

#### FR-2.4 热点区域聚类

- **需求描述**：基于DBSCAN或HDBSCAN算法，对出行起点/终点进行空间聚类
- **参数**：聚类半径 200–500m，最小样本数 10–50
- **输出**：Top-20 出行热点区域及其标签（如"深圳北站""福田CBD"等）

### 4.4 模块三：出行特征挖掘

#### FR-3.1 行程时长与距离分布

- **需求描述**：统计所有行程的时长分布和距离分布，拟合其概率密度函数
- **统计指标**：均值、中位数、标准差、偏度、峰度
- **可视化**：直方图 + 核密度估计曲线

#### FR-3.2 出行目的推断（可选）

- **需求描述**：基于OD点周边POI（兴趣点）信息，推断出行目的类别
- **目的分类**：通勤、购物、就医、休闲娱乐、接送等
- **推断规则**：
  - 早高峰住宅区→办公区：通勤
  - 办公区→商业区（午间）：餐饮/购物
  - 其他时段随机出行：休闲/个人事务

#### FR-3.3 车辆运营效率分析

- **需求描述**：分析每辆车的运营效率指标
- **指标列表**：
  - 日均载客次数
  - 日均载客里程/时长
  - 空驶率（空车时间/总运营时间）
  - 日均收入估算（基于里程和时长）
- **输出**：车辆运营效率排行榜

#### FR-3.4 跨区域出行模式

- **需求描述**：分析不同行政区之间的出行流量，识别职住分离现象
- **分析内容**：
  - 行政区级OD流量矩阵
  - 净流入/流出区域识别
  - 早晚高峰流向对比

### 4.5 模块四：可视化展示

#### FR-4.1 交互式数据看板

- **需求描述**：构建一个集成的数据可视化看板
- **组件清单**：
  1. 深圳地图 + 实时热力图层（可按时段切换）
  2. 出行量时序折线图（24小时）
  3. 行程时长分布直方图
  4. Top-10 热点区域柱状图
  5. 关键指标卡片（总出行量、平均时长、高峰时段等）
  6. OD流向弦图/桑基图

#### FR-4.2 分析报告自动生成

- **需求描述**：一键生成完整的分析报告
- **输出格式**：HTML / PDF
- **报告结构**：数据概况 → 清洗过程 → 时空分析 → 特征挖掘 → 结论与建议

## 5. 分析指标体系

### 5.1 指标分类框架

```
分析指标体系
├── 基础统计指标
│   ├── 总出行量
│   ├── 活跃车辆数
│   ├── 平均行程时长
│   └── 平均行程距离
├── 时间维度指标
│   ├── 小时出行量分布
│   ├── 高峰小时系数 (PHF)
│   ├── 昼夜出行比
│   └── 时段载客率
├── 空间维度指标
│   ├── 出行起点密度分布
│   ├── 出行终点密度分布
│   ├── 区域OD强度
│   └── 热点区域覆盖率
├── 运营效率指标
│   ├── 单车日均载客次数
│   ├── 空驶率
│   ├── 有效载客里程占比
│   └── 单车日均收入估算
└── 出行模式指标
    ├── 短途出行占比 (<3km)
    ├── 中长途出行占比 (>10km)
    ├── 跨区出行比例
    └── 通勤出行比例
```

### 5.2 核心KPI定义

| KPI | 计算公式 | 目标值 | 数据来源 |
|:---|:---|:---|:---|
| 日均出行总量 | `COUNT(行程记录)` | 464,718 | taxi_sz.csv |
| 单车日均载客次数 | `出行总量 / 车辆数` | ≈ 42 次 | taxi_sz.csv |
| 平均行程时长 | `AVG(Etime - Stime)` | 16.9 分钟 | taxi_sz.csv |
| 运营高峰时段 | `出行量 > 日均小时均值的 1.5 倍` | 早 7-9 时, 晚 17-19 时 | 分析产出 |
| 空驶率 | `空车时长 / 总运营时长` | < 40% | 乘车出行数据源.csv |
| 热点覆盖区域 | `Top-20 热点区域的出行占比` | > 60% | 分析产出 |

| 方案 | 优势 | 劣势 | 推荐场景 |
|:---|:---|:---|:---|
| **Folium + OpenStreetMap** | Python原生，静态导出方便 | 交互性有限 | Jupyter分析环境 |
| **Plotly + Mapbox** | 交互丰富，支持3D | 需Mapbox Token | Web报告 |
| **Deck.gl / Kepler.gl** | 大数据渲染性能好 | 学习成本高 | 大规模数据探索 |
| **ECharts + 百度/高德地图** | 中文生态完善 | 需地图API Key | 国内项目 |

> **推荐方案**：优先使用 **Folium** 进行快速探索分析，使用 **Kepler.gl** 进行交互式大数据可视化。
